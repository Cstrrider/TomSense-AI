# TomSense AI

Note: This is a personal project with a somewhat silly goal: Use claude code (mostly Opus audited by Fable 5) to produce its own repalcement entirely using Vibe Coding. I understand that is not for everyone. Even the rest of this description is written by Claude:

Self-hosted AI assistant for a CPU-only homelab — chat, agentic coding,
image generation, voice, RAG, and durable memory, with **all inference on
Cloudflare Workers AI** (free tier: 10k neurons/day). No GPU required.

- **Chat** with tool-calling: web search, page fetch, deep research, image
  generation/editing (Flux, or premium models via AI Gateway), code
  interpreter, semantic search over your uploaded documents, and
  per-user memory the assistant maintains itself.
- **Code mode**: an autonomous coding agent working in a hardened sandbox
  container, with live bind-mounts of your real source trees, diff
  approval, automatic post-edit verification, subagents, and optional
  one-click deploys of whitelisted projects.
- **Multi-user** behind Cloudflare Access (or single-user on localhost).
- **Android app** (Capacitor) that wraps your instance, with device tools —
  calendar, reminders, health, location, media — the assistant can call.
- **MCP both ways**: use remote MCP servers as extra tools, and expose your
  instance as an MCP server for Claude Code / claude.ai.

## Quickstart

Requirements: Docker with compose v2, a Cloudflare account (free), and an
API token with the *Workers AI* template.

```bash
git clone https://github.com/<you>/tomsense-ai && cd tomsense-ai
bash setup.sh            # asks: CF account, owner email, localhost-or-domain
docker compose up -d --build
# → http://localhost:8002
```

Optional capability profiles (any combination):

```bash
docker compose --profile voice --profile rag --profile jupyter --profile docker up -d --build
```

| Profile | Adds | Containers |
|---|---|---|
| *(none)* | chat, tools, code mode, web search | postgres, backend, frontend, sandbox, searxng |
| `voice` | 🎤 speech-to-text + 🔊 text-to-speech | whisper, piper |
| `rag` | document upload → semantic search | qdrant |
| `jupyter` | code_interpreter for regular chats | jupyter |
| `docker` | one-click sandbox mount apply + deploy_project | docker-socket proxy |

Everything degrades gracefully when a profile is off.

## Serving on a domain

Localhost mode trusts everyone as the owner — **never** expose it directly.
For internet access, put the frontend behind a Cloudflare Tunnel with
Cloudflare Access in front (any proxy that injects an authenticated-email
header also works). Steps in [CF_ACCESS_SETUP.md](CF_ACCESS_SETUP.md);
`setup.sh` flips the enforcement flag when you choose domain mode.

## Android app

The APK is a thin WebView shell over your instance plus native device
plugins (calendar, reminders, alarms, health, location, media control,
save-to-gallery). **The server URL is a runtime setting**: on first launch
the app asks for your instance URL, and you can change it any time in
Settings → App — so one APK works for any instance.

Build it with the *Android APK* GitHub Actions workflow (download the
artifact, sideload it), or locally:
`cd frontend && npx cap sync android && cd android && ./gradlew assembleDebug`.
Optionally set `TOMSENSE_SERVER_URL` (env var locally, repo variable for
the workflow) to pre-bake a default URL and skip the first-run prompt.

Because the app loads the live site, web UI changes ship instantly — the
APK only needs rebuilding when the native shell changes.

## Architecture

```
 Cloudflare Access (edge auth)          ── or ──   http://localhost:8002
        │  Cf-Access-Authenticated-User-Email
        ▼
 tomsense-frontend   nginx + SvelteKit SPA; proxies API → backend
        ▼
 tomsense-backend    FastAPI: agentic chat loop, tools, RAG, auth,
        │            provider abstraction (CF / Anthropic / any
        │            OpenAI-compatible endpoint you add in Settings)
        ├── tomsense-postgres   chats, users, memory, encrypted API keys
        ├── tomsense-sandbox    code-mode executor (hardened container)
        ├── tomsense-searxng    web_search
        └── profiles: whisper/piper (voice), qdrant (rag), jupyter,
                      docker-socket proxy (mount apply / deploys)
```

Key design points:

- **Bring-your-own models.** Defaults run on CF Workers AI; add Anthropic
  or any OpenAI-compatible provider (OpenRouter, Groq, local Ollama…) in
  Settings and pick models per tool or per chat.
- **Sandboxed coding.** The agent's shell lives in a read-only-rootfs,
  non-root, capability-dropped container with no docker socket. Deploys
  go through a separate socket proxy limited to container restarts, gated
  by mandatory per-deploy approval, against a server-side whitelist
  (`deploy-targets.json`).
- **Stored API keys are Fernet-encrypted at rest** (key generated by
  `setup.sh`).

## Ops notes

- Update: `git pull && docker compose up -d --build`
- Logs: `docker compose logs -f tomsense-backend`
- The backend prints `SECURITY:` warnings at startup for weak config —
  a clean start has none.
- Deeper install/security detail: [SETUP.md](SETUP.md)

## Layout

```
backend/app/     FastAPI: main.py (routes), chat.py (agent loop), tools*.py,
                 providers.py, rag.py, db.py, code_hints.py, mounts.py, …
frontend/        SvelteKit SPA + nginx; android/ Capacitor shell
sandbox/         code-mode executor image (see HARDENING.md)
searxng/         bundled meta-search config
docker-compose.yml  the whole stack (profiles for optional parts)
setup.sh         interactive first-run configuration → .env
```
